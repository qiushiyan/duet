# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What

`duet` — personal semi-AFK orchestrator for one developer's two-agent workflow: a read-only LLM **orchestrator** routes a snippet protocol between an **implementer** and a **reviewer**, inside a code-enforced statechart whose human gates agents cannot cross. Repo = design docs (authoritative for *what* to build) + implementation at root (pnpm + TS, no build step in dev).

Product goals — the bar every change is measured against:

- **Augment, never lock in.** Same CLIs, same snippets, standard transcripts and normal branches; manual takeover and resume must always work; the state file is a hint, never an obstacle.
- **The human owns substance.** The orchestrator does triage, never opinions; product/direction/environment questions always reach the human; gates are structural (statechart), not prompt-enforced.
- **Semi-AFK.** Walk away at plan approval; return to a ship packet or a well-formed queued question. Nothing runs between quiescent stops — no daemon.
- **Personal tool first, publish-ready.** Project knowledge enters only via the framing turn; the only config is role→provider bindings; exactly two providers. Shipped artifacts (`skills/`, README) are written for any user; no roadmap-by-request.

Status: full arc implemented (`new` → FRAME → SPEC → PLAN → AFK IMPL → Ship gate → DOCS → PR → opened PR), plus the concierge package (mid-phase `steer`, `status --json`, the shipped `skills/duet-concierge/`). FRAME→Ship live-verified (a scratch run + the first real planlab run, ~$93 claude-side); docs/pr/open phases uncrossed; gate pre-authorization awaits its first overnight run (Q20); the concierge package awaits its first live remote session. An opt-in interactive-Claude implementer transport (flat-quota billing) is built as a spike, awaiting its live-auth gate (`docs/interactive-transport.md`). Claude-Code-as-orchestrator is built over the attended arc (`duet orchestrate` / `new --interactive`): the human's interactive CC session drives FRAME → SPEC → PLAN and hands off to the headless driver at plan approval, on a host-neutral kernel proven at stdio-MCP parity (`docs/future-directions.md` §A). Unit/integration-verified (418 tests); the live end-to-end against a real Claude Code session is deferred to its auth gate and has not run, and the environment smoke tests are pending. The concierge **run-operations** layer — `duet doctor` health + connectivity, the health-enriched heartbeat, the signal-only `humanDecisions` / `cause`+`errorClass` triage fields, `status --brief`, the hardened non-TTY write path, and opt-in bounded infra retry — is implemented and test-verified, live verification + smoke tests pending (the codex error-envelope branch is validated synthetically only). Codex-as-orchestrator deliberately unbuilt (Q17).

## Commands

- `pnpm typecheck` — checker-only tsc. No build in dev: Node 24 runs `.ts` directly. Erasable syntax only (no enum/namespace/param-properties); explicit `.ts` import extensions.
- `pnpm build` — publish-only tsdown bundle (`dist/cli.mjs`; `prepack` chains typecheck → tests → build, and `publishConfig.bin` swaps the published bin to it). Dev and the global `duet` link (`pnpm add -g .`) always run `src/cli.ts`; npm-installed copies can't (Node refuses `.ts` in `node_modules`). Details: `docs/engineering.md` §Build & publish.
- `pnpm test` — the Vitest behavior suite (`tests/`, standalone; fixtures in `tests/helpers/`).
- `node src/cli.ts` — the CLI: `new [--spec][--framing][--template][--gates-at <phases|overnight|skip-plan>][--retry-infra <n>][--tmux][--interactive]` (bare = $EDITOR on a template-seeded `.duet/framing-draft.md`; `--template <name>` seeds from project `.duet/templates/<name>.md`, bare prefers `.duet/templates/default.md`; `--interactive` orchestrates from your own interactive Claude Code session instead of the headless driver) / `orchestrate [runId]` (bring up — or reconnect — the interactive orchestrator over FRAME → PLAN) / `continue [--approve ["rider"]|--reject ["…"]|--answer "…"][--headless]` (an approval may carry a rider; at a TTY a bare `--approve`/`--reject`/`--answer` opens $EDITOR; off a TTY a bare `--approve` is no-rider and a bare `--reject`/`--answer` fails fast, with `--reject-file`/`--answer-file` (or `-` for stdin) for verbatim text past shell quoting; on an interactive run it advances the machine inline, and `--headless` drops the run to the detached driver) / `steer "<note>"` (the mid-phase human voice — live or crashed phases only) / `status [--json][--brief][--wait]` / `doctor [--json]` / `runs` / `view` / `logs` / `takeover <role>` / `abandon [--purge]` (kill a run's driver and mark it abandoned — reversible; `--purge` also deletes its session transcripts). `duet --help` carries the run model (the CLI is self-documenting — the concierge skill leans on it). Commands return immediately; phases run in a detached `_drive` child (stdout → `.duet/runs/<id>/driver.log`, pid-guarded); with pre-authorized gates the driver lives through the whole stretch to the next attended stop. An interactive run has no `_drive` during the attended arc — the interactive orchestrator session drives each phase and `duet continue` crosses inline — until the plan-gate handoff.

## Map

- `src/phases.ts` — **the phase table**: arc order + every per-phase fact (gate names/copy, round caps, budgets, timeouts, snippet sets). The statechart and all consumers derive from it.
- `src/harness/` — `machine.ts` (statechart: a phase emits an internal `phase.*` event; gates cross only on `human.*` — two disjoint vocabularies; `interactiveMachine` is the inert-`phaseDriver` variant the interactive host rests on), `phase-events.ts` (that vocabulary + the pure, phase-matched marker→event read), `tools.ts` (the 8 orchestrator tools — incl. `get_task`, the interactive host's brief surface — as a host-neutral `KernelTool` registry + protocol rails, incl. the post-terminal quiescence rail and the per-phase rails seam; every phase-continuing result also delivers pending steers; the `activeTurns` hint + health-enriched heartbeat; `advance_phase`'s signal-only `human_decisions`), `driver.ts` (the in-process host: Agent SDK session per phase, resolves the `phase.*` event, steer carry-forward; infra classify + opt-in bounded retry; honors a post-call terminal marker; `toSdkTools` adapter), `mcp-server.ts` (the stdio-MCP adapter + the hidden `duet _mcp` body: a single-phase server, plus the run-scoped phase-less server the interactive orchestrator connects to — resolves the live phase per call, caches rails-only) + `stdio-host.ts` (the out-of-process driver + the `Orchestrate` seam — the same kernel at parity), `lifecycle.ts` (detached driver + `killDriver` + `gates_at` auto-cross + the spent-marker guard + `probeRunPosition` — where a run is: running/interactive/gate/flag/crashed/abandoned/done, derived from driver liveness + parked snapshot + state evidence, never the snapshot alone — plus `crossInteractive`, the inline interactive crossing), `orchestrator-prompts.ts`.
- `src/providers/` — the worker seam: contract, claude + codex adapters, factory; the claude provider has two transports — headless `claude -p` and the opt-in interactive TUI driver (`interactive-claude.ts` parser/locator/worker + the `pane.ts` `PaneController` injection seam; `docs/interactive-transport.md`). `src/orchestrate.ts` — the `duet orchestrate` launcher: builds the `claude` argv (the run-scoped `_mcp` server self-referenced off the running CLI, the `skills/duet/identity.md` system prompt, the single gate-safety `ask` rule, and a seeded kickoff prompt so the session opens working, not blank) over the injectable `ClaudeLauncher` spawn seam; marks the run `orchestrationHost='interactive'`, restoring prior state on a failed launch. `src/run-store.ts` — run-dir persistence (atomic), the CLI↔driver input-staging handshake, the steer store (file-per-steer, rename-consume), the interactive-host markers (`orchestrationHost`, `orchestratorCostPartial`), and run teardown (`duet abandon`: the abandoned marker + `--purge`). `src/sessions.ts` — locating a run's provider session transcripts by id; the one module reaching outside `.duet/` (what `--purge` deletes), plus `resolveSessions` (the `status --json .sessions[]` map) and `readRoleTranscriptTail`. `src/worker-health.ts` — the pure health substrate (taxonomy + `probeRole` + the shared `currentTerminalError` rule + `retryDecision`; imports no fs/lifecycle/status, by the cycle constraint). `src/doctor.ts` — `duet doctor`'s composer/renderer + connectivity probe (imported only by `cli.ts`). `src/framing.ts` — framing-draft seed selection (built-in template or project `.duet/templates/`), editor flow, frontmatter boundary. `src/status.ts` — the status model + its two renderers (human text, `status --json` verbatim; schema additive-only), incl. the interactive stop arm; plus `status --brief` and the additive `sessions[]` / `humanDecisions` / `cause`+`errorClass` fields. `src/cli.ts` — wiring only; parses under `import.meta.main`, so the command table is importable. View glue: `colorize.ts`, `tmux-view.ts`, `notify.ts`.
- `skills/duet-concierge/` — the shipped concierge skill (a Claude Code session as duet's remote layer). `skills/duet-frame/` — the shipped framing-author skill: turns a rough problem into a run framing (sharpen wording/names/structure + set gate posture, never change intent or propose solutions), then emits the `duet new --interactive` command. `skills/duet/` — the orchestrator identity (`identity.md`, fed as the launcher's system prompt — not a slash command). All three pinned to the CLI by `tests/skill.test.ts`.
- Run state `.duet/runs/<id>/`: `state.json` is a hint — the 3 provider JSONL transcripts are truth. `steers/` holds the staged mid-phase notes (`delivered/` = consumed). `notes.md` = dogfooding journal (Q13/Q19/Q20 evidence).

Invariants that bite if forgotten (full reasoning: `docs/engineering.md`, `docs/open-questions.md`): the `ask_human` pause is cooperative — mechanical SDK pauses corrupt resume (repros in `src/spike/`); claude workers compact via a literal `/compact` prompt, codex auto-compacts and must never be sent a command; one branch per run, fixed before the first worker prompt; worker budget is per-turn and must never shape scope (Q19); steers live in `steers/`, never `state.json` — a CLI write there would race a live driver's saves; log files stay plain text — color is view-time only; runtime artifacts live under self-ignored `.duet/`, never the repo root; gate-crossing is un-forgeable by vocabulary — a phase emits `phase.*`, only `human.*` crosses a gate, and no tool emits `human.*` (in-process, over the stdio MCP boundary, or from a Bash-equipped interactive orchestrator session); the interactive orchestrator's one safety setup is a single `ask` rule on `duet continue` that survives the human's bypass-permissions default, so a gate still takes a human tap; the persisted terminal marker is the cross-process phase decision, cleared deliver-before-clear with a spent-marker guard keyed off the restored snapshot, so a crash can't replay a stale decision over a human's answer/reject; the health substrate stays pure (`worker-health.ts` imports no fs/lifecycle/status, else a `status → worker-health → lifecycle → status` value cycle closes — the fs tail-read lives in `sessions.ts`, the lifecycle-needing composition in `doctor.ts`, imported only by `cli.ts`), and infra failures classify on every surface but auto-retry only on the headless driver (a human is present on the interactive host).

## Conventions

- **Docs lead, code follows.** Code/docs disagreement = doc bug or design regression; resolve explicitly, never silently. How docs are kept lives in `docs/documentation-standards.md`.
- **Evidence-backed claims.** Workflow claims cite `examples/*.jsonl` turns or run logs; tag **(observed)** vs **(general)**.
- **Tests are behavior-through-interface.** Fake only at the six seams (`docs/engineering.md` §Seams); never mock our own modules.
- **Personal tool / augmentation** — see the product goals above; they are conventions too.
