# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What

`duet` — personal semi-AFK orchestrator for one developer's two-agent workflow: a read-only LLM **orchestrator** routes a snippet protocol between an **implementer** and a **reviewer**, inside a code-enforced statechart whose human gates agents cannot cross. It is workflow-aware: a run picks one arc — the thorough `full` (spec → plan → … → PR) or the lighter `rir` (research → implement → ship). Repo = design docs (authoritative for *what* to build) + implementation at root (pnpm + TS, no build step in dev).

Product goals — the bar every change is measured against:

- **Augment, never lock in.** Same CLIs, same snippets, standard transcripts and normal branches; manual takeover and resume must always work; the state file is a hint, never an obstacle.
- **The human owns substance.** The orchestrator does triage, never opinions; product/direction/environment questions always reach the human; gates are structural (statechart), not prompt-enforced.
- **Semi-AFK.** Walk away at the workflow's handoff gate (full: plan approval; rir: Direction); return to a ship packet or a well-formed queued question. Nothing runs between quiescent stops — no daemon.
- **Personal tool first, publish-ready.** Project knowledge enters only via the framing turn; the only config is role→provider bindings; exactly two providers. Shipped artifacts (`skills/`, README) are written for any user; no roadmap-by-request.

Status: full arc implemented (`new` → FRAME → SPEC → PLAN → AFK IMPL → Ship gate → DOCS → PR → auto-opened PR), plus the concierge package (mid-phase `steer`, `status --json`, the shipped `skills/duet-concierge/`). The workflow registry + the second arc, **rir** (research → Direction → implement → Ship, no spec/plan/docs/PR; `afk` preset), are implemented and unit/integration-verified (594 tests); no live RIR run has executed end-to-end (deferred to the same auth gate as the interactive orchestrator) and the human's environment smoke tests are pending. FRAME→Ship live-verified (a scratch run + the first real planlab run, ~$93 claude-side); docs/pr/open phases uncrossed; gate pre-authorization awaits its first overnight run (Q20); the concierge package awaits its first live remote session. An opt-in interactive-Claude implementer transport (flat-quota billing) is built as a spike, awaiting its live-auth gate (`docs/interactive-transport.md`). Claude-Code-as-orchestrator is built over the attended arc (`duet orchestrate` / `new --interactive`): the human's interactive CC session drives FRAME → SPEC → PLAN and hands off to the headless driver at plan approval, on a host-neutral kernel proven at stdio-MCP parity (`docs/future-directions.md` §A). On the interactive host `send_prompt` is **fire-and-collect** — it dispatches the worker turn and returns at once (collect with `check_turns`), so the session stays live while a turn runs; a single-writer lease and an on-disk orphan/`takeover` reconnect contract back it (`docs/specs/2026-06-21-async-interactive-send-prompt.md`). Unit/integration-verified (594 tests, the whole suite); the live end-to-end against a real Claude Code session is deferred to its auth gate and has not run, and the environment smoke tests are pending. The concierge **run-operations** layer — `duet doctor` health + connectivity, the health-enriched heartbeat, the signal-only `humanDecisions` / `cause`+`errorClass` triage fields, `status --brief`, the hardened non-TTY write path, and opt-in bounded infra retry — is implemented and test-verified, live verification + smoke tests pending (the codex error-envelope branch is validated synthetically only). The **rails-and-UX bundle** makes the rails opt-in with friction-free defaults — the full arc's PR auto-opens by default (opt-in pre-open stop via `gates_at: pr`), `duet afk` hands a run off mid-session from any interactive gate, per-turn budget caps are opt-in/off-by-default and a hit cap degrades to a resumable checkpoint rather than crashing, plus the UX cluster (provider-agnostic onboarding, `write_note` at a gate, the per-turn cost/context footer, a truthful state label) — implemented and unit/integration-verified (594 tests), with the environment smoke tests and a live interactive→headless `duet afk` handoff still pending (auth-gated). Codex-as-orchestrator deliberately unbuilt (Q17).

## Commands

- `pnpm typecheck` — checker-only tsc. No build in dev: Node 24 runs `.ts` directly. Erasable syntax only (no enum/namespace/param-properties); explicit `.ts` import extensions.
- `pnpm build` — publish-only tsdown bundle (`dist/cli.mjs`; `prepack` chains typecheck → tests → build, and `publishConfig.bin` swaps the published bin to it). Dev and the global `duet` link (`pnpm add -g .`) always run `src/cli.ts`; npm-installed copies can't (Node refuses `.ts` in `node_modules`). Details: `docs/engineering.md` §Build & publish.
- `pnpm test` — the Vitest behavior suite (`tests/`, standalone; fixtures in `tests/helpers/`).
- `node src/cli.ts` — the CLI: `new [--workflow <full|rir>][--spec][--framing][--template][--gates-at <phases|preset>][--retry-infra <n>][--budget <off|default|N>][--tmux][--interactive]` (bare = $EDITOR on a template-seeded `.duet/framing-draft.md`; `--workflow` picks the arc, default `full`, also a `workflow:` framing key; `--gates-at` presets are workflow-scoped — full: `overnight`/`skip-plan`, rir: `afk`; `--template <name>` seeds from project `.duet/templates/<name>.md`, bare prefers `.duet/templates/default.md`; `--interactive` orchestrates from your own interactive Claude Code session instead of the headless driver) / `orchestrate [runId]` (bring up — or reconnect — the interactive orchestrator over the attended arc up to the handoff gate; full: FRAME → PLAN, rir: RESEARCH → Direction) / `continue [--approve ["rider"]|--reject ["…"]|--answer "…"][--edit][--headless]` (an approval may carry a rider — inline, or composed in $EDITOR with `--approve --edit`; a bare `--approve` approves with no rider. A bare `--reject`/`--answer` opens $EDITOR at a TTY and fails fast off one, with `--reject-file`/`--answer-file` (or `-` for stdin) for verbatim text past shell quoting; on an interactive run it advances the machine inline, and `--headless` drops the run to the detached driver) / `steer "<note>"` (the mid-phase human voice — live or crashed phases only) / `afk [preset] [runId]` (one-tap mid-session handoff from an interactive gate — re-set the downstream gate posture, bare = attend none, then drop to the headless driver) / `status [--json][--brief][--wait]` / `doctor [--json]` / `runs` / `view` / `logs` / `takeover <role>` / `abandon [--purge]` (kill a run's driver and mark it abandoned — reversible; `--purge` also deletes its session transcripts). `duet --help` carries the run model (the CLI is self-documenting — the concierge skill leans on it). Commands return immediately; phases run in a detached `_drive` child (stdout → `.duet/runs/<id>/driver.log`, pid-guarded); with pre-authorized gates the driver lives through the whole stretch to the next attended stop. An interactive run has no `_drive` during the attended arc — the interactive orchestrator session drives each phase and `duet continue` crosses inline — until the workflow's handoff gate (full: plan; rir: Direction).

## Map

- `src/phases.ts` — **the workflow registry** (`WORKFLOWS`): the arcs as data — each arc's ordered phases, entry route, handoff gate, gate presets, force-attended gates + the inverse default-pre-authorized set (materialized into a run's `gatesAt` at creation) + every per-phase fact (gate names/copy, round caps, budgets, timeouts, snippet sets). The single source: flat lookups (`PHASE`) derive on globally-unique phase names, topology/policy through workflow-scoped helpers (`machineFor`, `phasesOf`, `gatePhasesOf`, `phaseOfGateState`, `entryOf`); `validateRegistry` guards the derivation invariants at load.
- `src/harness/` — `machine.ts` (statechart, built per-arc by `machineFor(workflow)`: a phase emits an internal `phase.*` event; gates cross only on `human.*` — two disjoint vocabularies; `interactiveMachineFor(workflow)` is the inert-`phaseDriver` variant the interactive host rests on), `phase-events.ts` (that vocabulary + the pure, phase-matched marker→event read), `tools.ts` (the 8 base orchestrator tools — incl. `get_task`, the interactive host's brief surface — plus `check_turns` on the interactive host, as a host-neutral `KernelTool` registry + protocol rails, incl. the post-terminal quiescence rail, the async phase-exit gate, and the per-phase rails seam; `send_prompt` is host-switched, blocking headless vs dispatch-to-background interactive; every phase-continuing result also delivers pending steers; the `activeTurns` hint + health-enriched heartbeat; `advance_phase`'s signal-only `human_decisions`), `turn-dispatcher.ts` (the interactive host's pending-turn engine: dispatch → settle → collect over the `WorkerProvider` promise seam, non-throwing/total, lease-fenced), `driver.ts` (the in-process host: Agent SDK session per phase, resolves the `phase.*` event, steer carry-forward; infra classify + opt-in bounded retry; honors a post-call terminal marker; `toSdkTools` adapter), `mcp-server.ts` (the stdio-MCP adapter + the hidden `duet _mcp` body: a single-phase server, plus the run-scoped phase-less server the interactive orchestrator connects to — resolves the live phase per call, caches rails-only, holds the single-writer lease `mcp-owner.json`) + `stdio-host.ts` (the out-of-process driver + the `Orchestrate` seam — the same kernel at parity), `lifecycle.ts` (detached driver + `killDriver` + `gates_at` auto-cross + the spent-marker guard + `probeRunPosition` — where a run is: running/interactive/gate/flag/crashed/abandoned/done, derived from driver liveness + parked snapshot + state evidence, never the snapshot alone — plus `crossInteractive`, the inline interactive crossing, and `waitForTurnOrStop`, the turn-aware `status --wait`, and `enterAfk`, the mid-session AFK handoff), `orchestrator-prompts.ts`.
- `src/providers/` — the worker seam: contract, claude + codex adapters, factory; the claude provider has two transports — headless `claude -p` and the opt-in interactive TUI driver (`interactive-claude.ts` parser/locator/worker + the `pane.ts` `PaneController` injection seam; `docs/interactive-transport.md`). `src/orchestrate.ts` — the `duet orchestrate` launcher: builds the `claude` argv (the run-scoped `_mcp` server self-referenced off the running CLI, the `prompts/orchestrator-identity.md` system prompt, the single gate-safety `ask` rule, and a seeded kickoff prompt so the session opens working, not blank) over the injectable `ClaudeLauncher` spawn seam; marks the run `orchestrationHost='interactive'`, restoring prior state on a failed launch. `src/run-store.ts` — run-dir persistence (atomic), the CLI↔driver input-staging handshake, the steer store (file-per-steer, rename-consume), the interactive-host markers (`orchestrationHost`, `orchestratorCostPartial`), the interactive pending-turn lifecycle (`pendingTurns` + the one-way `workerDispatched` flag) and the single-writer lease (`mcp-owner.json`: `acquireMcpOwner`/`holdsMcpOwner`), the run-state policy resolvers `gateAttended`/`budgetFor` (the frozen opt-in `budget`) and the one mutable-posture write `setGatesAt`, and run teardown (`duet abandon`: the abandoned marker + `--purge`). `src/sessions.ts` — locating a run's provider session transcripts by id; the one module reaching outside `.duet/` (what `--purge` deletes), plus `resolveSessions` (the `status --json .sessions[]` map) and `readRoleTranscriptTail`. `src/worker-health.ts` — the pure health substrate (taxonomy + `probeRole` + the shared `currentTerminalError` rule + `retryDecision`; imports no fs/lifecycle/status, by the cycle constraint). `src/doctor.ts` — `duet doctor`'s composer/renderer + connectivity probe (imported only by `cli.ts`). `src/framing.ts` — framing-draft seed selection (built-in template or project `.duet/templates/`), editor flow, frontmatter boundary. `src/status.ts` — the status model + its two renderers (human text, `status --json` verbatim; schema additive-only), incl. the interactive stop arm; plus `status --brief` and the additive `sessions[]` / `humanDecisions` / `cause` (incl. `budget`)+`errorClass` / `pendingTurns[]` fields and the honest `displayState` label. `src/cli.ts` — wiring only; parses under `import.meta.main`, so the command table is importable. View glue: `colorize.ts`, `tmux-view.ts`, `notify.ts`.
- `skills/duet-concierge/` — the shipped concierge skill (a Claude Code session as duet's remote layer). `skills/duet-frame/` — the shipped framing-author skill: turns a rough problem into a run framing (sharpen wording/names/structure + set gate posture, never change intent or propose solutions), then emits the `duet new --interactive` command. `prompts/orchestrator-identity.md` — the orchestrator identity, fed as the launcher's system prompt (not a skill, not a slash command). All three are pinned to the CLI by `tests/skill.test.ts`.
- Run state `.duet/runs/<id>/`: `state.json` is a hint — the 3 provider JSONL transcripts are truth. `steers/` holds the staged mid-phase notes (`delivered/` = consumed). `notes.md` = dogfooding journal (Q13/Q19/Q20 evidence).

Invariants that bite if forgotten (full reasoning: `docs/engineering.md`, `docs/open-questions.md`): the `ask_human` pause is cooperative — mechanical SDK pauses corrupt resume (repros in `src/spike/`); claude workers compact via a literal `/compact` prompt, codex auto-compacts and must never be sent a command; one branch per run, fixed before the first worker prompt; worker budget is per-turn, opt-in and off by default (off ≡ absent, never `0`), must never shape scope (Q19), and a hit cap is a resumable checkpoint, not an infra crash; steers live in `steers/`, never `state.json` — a CLI write there would race a live driver's saves; log files stay plain text — color is view-time only; runtime artifacts live under self-ignored `.duet/`, never the repo root; gate-crossing is un-forgeable by vocabulary — a phase emits `phase.*`, only `human.*` crosses a gate, and no tool emits `human.*` (in-process, over the stdio MCP boundary, or from a Bash-equipped interactive orchestrator session); `gates_at` is the complete attend set, not a delta (so `--gates-at pr` attends only the Open-PR gate), and the full arc's Open-PR gate auto-opens the PR by default yet an attended gate still takes a human tap; the interactive orchestrator's one safety setup is a single `ask` rule on `duet continue` that survives the human's bypass-permissions default, so a gate still takes a human tap; the persisted terminal marker is the cross-process phase decision, cleared deliver-before-clear with a spent-marker guard keyed off the restored snapshot, so a crash can't replay a stale decision over a human's answer/reject; the health substrate stays pure (`worker-health.ts` imports no fs/lifecycle/status, else a `status → worker-health → lifecycle → status` value cycle closes — the fs tail-read lives in `sessions.ts`, the lifecycle-needing composition in `doctor.ts`, imported only by `cli.ts`), and infra failures classify on every surface but auto-retry only on the headless driver (a human is present on the interactive host); the interactive host's `send_prompt` is fire-and-collect, fenced by a single-writer lease (`mcp-owner.json`) at two boundaries — the synchronous tool gate and the background settle — so a superseded server writes nothing, and its background turn lifecycle is non-throwing/total (a faulting turn flips to a collectible `failed`, never strands a role `running` — which would hang `check_turns`/`status --wait` under AFK); a turn orphaned by a session quit lives as a durable on-disk record (never a transcript guess) that refuses a same-role resend and blocks phase-exit until `duet takeover` clears it, and the lease lands before the async writer with the orphan invariant holding on disk from its first commit, so every committed tree is self-safe under AFK.

## Conventions

- **Docs lead, code follows.** Code/docs disagreement = doc bug or design regression; resolve explicitly, never silently. How docs are kept lives in `docs/documentation-standards.md`.
- **Evidence-backed claims.** Workflow claims cite `examples/*.jsonl` turns or run logs; tag **(observed)** vs **(general)**.
- **Tests are behavior-through-interface.** Fake only at the six seams (`docs/engineering.md` §Seams); never mock our own modules.
- **Personal tool / augmentation** — see the product goals above; they are conventions too.
