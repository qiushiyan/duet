# Engineering notes — the codebase's mental model

How the implementation is shaped and why. Companion to `automation-design.md` (the system design: roles, phases, gates, policies). Read that before changing *behavior*; read this before moving *code*. Audience: an engineer — human or agent — touching `src/` for the first time.

## The trust gradient

One sentence holds the architecture: **deterministic code constrains an intelligent orchestrator, which commands two workers — trust flows one way.**

```
statechart (when)   →   tool handlers (what's allowed)   →   prompts (judgment)
hard guarantee          enforced rail                        steerable
```

Placement rule for any new rule: violating it would break the human's authority or the run's integrity → statechart or tool handler. Violating it would merely waste tokens or quality → prompt. The design history (automation-design §"Design history") is one long lesson about this line — the pre-pivot router put *judgment* in code (string-matching disagreements, caps as exit rules) and failed; the pivot moved judgment up into an LLM and moved *guarantees* down into structure. Never drift back: no code that parses worker prose for meaning, no prompt that is the only thing between an agent and a gate.

## Seams

A seam = a place behavior swaps without editing code in place. Six deliberate ones exist; tests fake **only** at these:

| Seam | Adapters | Why |
|---|---|---|
| `WorkerProvider` (`providers/types.ts`) | claude, codex, tests' `FakeWorker` | Role–provider decoupling (design) doubles as worker scripting (tests) |
| `RunOrchestratorTurn` (`harness/driver.ts`) | Agent SDK, scripted sessions | The in-process SDK boundary; a fake session receives the real tool handlers and may invoke them |
| `Orchestrate` (`harness/stdio-host.ts`) | scripted client (tests), interactive CC session | The out-of-process sibling of `RunOrchestratorTurn` — drives the kernel over a real stdio MCP boundary; the seam the interactive orchestrator runs through |
| `phaseDriver` actor (`harness/machine.ts`) | `runPhase`, `runPhaseOverStdio`, inert (`interactiveMachine`), `machine.provide` scripts | XState's own substitution mechanism; the in-process, stdio, and interactive hosts all plug in here, so the lifecycle is reused unchanged |
| Environment (`$EDITOR`, config path, notify fn, `ClaudeLauncher`, `isTTY`, connectivity `fetch`, transcript `home`) | real / stubbed / recording | OS boundaries — incl. the `duet orchestrate` process spawn (`orchestrate.ts`, a recording launcher in tests), the non-TTY write path, and the doctor connectivity / transcript reads |
| `PaneController` (`providers/pane.ts`) | `TmuxPane`, tests' `FakePane`, owned-pty (later) | The injection/process sub-seam *inside* the interactive claude transport — keeps the transcript parser transport-independent so owned-pty swaps in unchanged (`interactive-transport.md`) |

Rule: a new abstraction earns a seam only when a second adapter exists or a test needs one (`PaneController` earned both — a test now, pty later). Otherwise call directly — pass-through layers get deleted on sight.

## Patterns that carry the design

- **Workflow registry.** Adding or tuning a phase is one row in `phases.ts`; adding an *arc* is one entry in `WORKFLOWS` (mostly data). A `Record<PhaseName, …>` appearing anywhere else means the field belongs in the registry.
- **Flat vs workflow-scoped.** Pure per-phase lookups stay flat and global — legal because phase names are globally unique (`validateRegistry` enforces it). Topology and policy — which phases, in what order, which gate maps to which phase, the machine, the position probe — are workflow-scoped (`machineFor(workflow)`, `phasesOf`, `phaseOfGateState`). Putting a scoped concern in a flat lookup is the mistake to avoid; a phase accepted from outside (`_mcp`) is validated against the run's arc.
- **Role policy as data; worker vs voice.** A role's asymmetries live once in `roles.ts`'s table, never as `role === 'reviewer'` checks sprinkled across the hosts (the consultant added a third role and the table absorbed it — the deletion test: drop the table and the conditionals re-grow at N sites). The split that threads every enumeration surface: **workers** (the bound makers/reviewers — `workerRolesFor`) drive routing, send-prompt enums, branch-fixing, orphan recovery; **voices** (`voicesFor` = orchestrator + workers) drive display/log/health, which must keep the orchestrator a reviewer-routing surface would drop. Picking the wrong one is a silent bug — a voice surface routed through `workerRolesFor` loses the orchestrator; a worker surface through `voicesFor` tries to prompt it. And the optional consultant is **default-off byte-for-byte**: absent its binding, every surface (config, tool schema, the snippet library, the phase briefs) reads exactly as before — the run-facing guarantee, tested at the tool altitude, not just the helper's.
- **Rails as tool results.** Every rail is a handler that *refuses with steering text*: template-economy warn-once-then-allow, review-round backstop caps, branch-fixed-after-first-prompt, advance-needs-a-review-round. A rail isn't real until its result text tells the orchestrator what to do instead (prompting doc, convention 5).
- **Host-neutral kernel, three hosts.** The tool surface is one `KernelTool` registry (`tools.ts`), independent of any SDK; thin adapters host it — the in-process Agent SDK server (`driver.ts`'s `toSdkTools`), the standard single-phase stdio MCP server (`mcp-server.ts`, reached via `duet _mcp <runId> <phase>`), and the run-scoped phase-less server the interactive orchestrator connects to (`duet _mcp <runId>`). The same handlers, rails, and steer delivery serve all, so an out-of-process orchestrator behaves identically to the in-process one — proven by a parity test driving a real `_mcp` subprocess. The single dependency this added, `@modelcontextprotocol/sdk`, is pinned to the one version the Agent SDK already resolves, so every host shares one `CallToolResult` type.
- **Interactive orchestrator host.** The human's own CC session is an orchestrator host over the attended arc, up to the workflow's handoff gate (full: through PLAN; rir: through Direction). Its server is *run-scoped*, not phase-scoped: a long-lived session can't keep swapping a static phase arg, so it resolves the live phase from disk per call and refuses once the run is no longer interactively hosted (`orchestrationHost`), closing the two-writer gap against a handed-off headless driver. The **rails-only cache** is the load-bearing tactic: the tool surface is rebuilt per call against *fresh* disk state — so a cross-process `duet continue` (a same-phase reject/answer) is seen immediately, not served stale — while only the per-phase rails (the same-role in-flight guard, the warn-once set) and providers are cached and rebuilt at a phase boundary. The machine rests inert (`interactiveMachine`): a non-quiescent phase-loop snapshot is a legitimate rest, and `crossInteractive` advances it inline (consume the marker's `phase.*` to reach the gate/flag, *then* apply `human.*` — a phase loop ignores `human.*`, only the gate/flag takes it) with no `_drive` until the handoff gate. Gate safety is structural as everywhere (`get_task`/`advance_phase` only park), plus one `ask` rule on `duet continue` that survives the human's bypass-permissions default — the colon form `Bash(duet continue:*)`, matching the concierge's read-verb rules.
- **Fire-and-collect worker turns (interactive host).** A CC assistant turn can't yield while a `tool_use` is outstanding, so a blocking `send_prompt` would freeze the session for the minutes a worker runs. On the interactive host an `async` dep flips `send_prompt` from blocking to *dispatch-and-return* and exposes `check_turns` to collect later — a host switch in the one registry, not a fork (the headless driver keeps the blocking path and never leaves a turn in flight). The `turn-dispatcher.ts` engine owns the pending-turn lifecycle (`dispatch` → settle → `collectReady`), running the *same* settle/render/heartbeat steps a blocking call does, spread across time; it reads fresh disk state at each step because it outlives the dispatching call. Three invariants carry it: (1) a **single-writer lease** (`mcp-owner.json`) fenced at two boundaries — `toolsFor`'s synchronous gate covers every dispatch-time write, the background settle keeps its own non-throwing lease check — so a superseded server's late settle writes nothing; (2) the background lifecycle is **non-throwing / total** on every exit: a faulting setup, launch, or settle flips the role to a collectible `failed` rather than stranding it `running` (which would hang `check_turns` / `status --wait` forever under AFK), and the heartbeat is always stopped; (3) a turn orphaned by a session quit is a **durable record with no live owner** — detected on disk, never from a transcript guess — which refuses a same-role resend and blocks phase-exit until `duet takeover` clears it. The slice ordering is load-bearing: the lease lands before the async writer, and the orphan invariant holds on disk from the writer's first commit, so every committed tree is self-safe under AFK. Test-verified; the live end-to-end and environment smoke tests are still pending.
- **Cooperative pause.** `ask_human` persists the question at call time, the result text says "end your turn", the process exits at quiescence. Mechanical SDK interrupts corrupt resume — proven, with executable repros (`src/spike/repro-*.ts`).
- **Parallel worker turns.** The claude CLI runs MCP tools serially unless `annotations.readOnlyHint` is true (its scheduler's concurrency test for MCP tools — verified 2.1.175, undocumented; full note: `docs/prompting-and-tool-design.md` §Concurrency). `send_prompt` carries the annotation as a concurrency hint, not a purity claim, so independent turns to different workers overlap; the genuinely unsafe case — two concurrent turns into one session — is refused by the handler's same-role in-flight rail. If parallel sends stop overlapping after a CLI upgrade, re-verify that mapping first.
- **Staging handshake.** Human input crosses the CLI→driver process boundary via `stageHumanInput`/`consumeHumanInput`, consumed exactly once so a retried driver can't replay an answer. Known edge, chosen deliberately: a crash after consume loses the staged text — the crash question asks the human to re-supply, which beats risking double-delivery into a session that may have already used it.
- **Steer channel.** The human's mid-phase voice rides tool results: file-per-steer staging under `steers/` (never `state.json` — a CLI write would race a live driver's saves), delivery appended to every phase-continuing result, carry-forward through the next harness prompt otherwise. The crash trade inverts the staging handshake's: deliver-then-consume, so a crash *redelivers* rather than loses (a repeated instruction is benign where a lost one is not). Turn-ending results (advance recorded, question queued) never carry steers — guidance delivered into a dying turn lands and dies — and the delivery path is fail-soft: a steer bug must never corrupt a tool result.
- **Position probe.** Where a run is cannot be read from the snapshot alone — persistence is quiescence-only, so mid-phase the snapshot shows the *previous* stop. `probeRunPosition` joins driver liveness, the parked snapshot, and the state evidence the driver writes continuously (`phaseStarted`, `pendingQuestion`) into running / gate / flag / crashed / abandoned / done; a crashed position names how its dead crossing is re-uttered (`approve` / `answer`), so bare `duet continue` recovers every crash shape. The `abandoned` short-circuit comes first: `duet abandon` kills the driver and writes a marker, so a deliberate stop reads as abandoned rather than crashed — and stays reversible, since `continue` clears the marker and the transcripts are kept. Steer gating, crash recovery, and the status model all derive from this one probe.
- **Crash = flag.** Any infrastructure failure lands the run on an actionable queued question, never a silent state; a question the orchestrator already queued is never overwritten by a crash question. The stdio host owns this across the process boundary too: a dead `_mcp` peer becomes the same persisted question, so crash = flag survives the boundary, and a silent turn gets one nudge before flagging (parity with the in-process driver).
- **Infra classification & opt-in retry.** One pure rule decides "is this run broken right now": `currentTerminalError` — a classified terminal error that is recent *and* not superseded by later activity — shared by the `crashed` verdict and the driver's infra classification so the two can't drift. The retry *policy* is consulted in exactly one place, the headless driver (`retryDecision`); the classification *taxonomy* reaches two, the driver and the stdio interactive host — but only the headless path retries (a human is present on the interactive host, so it classifies and hands back). Default-off; transient classes back off and retry, `auth` retries once then escalates as `login-required`, the rest flag; exhaustion always flags.
- **Opt-in rails, safe defaults.** Two run-policy resolvers read run state against the registry and fall back to legacy behavior when the knob is absent: `gateAttended` (absent `gatesAt` ⇒ attend all) and `budgetFor` (absent `budget` ⇒ no caps; off ≡ absent, never `0`). The defaults are friction-free — the PR auto-opens (Open-PR pre-authorized by default), budget caps are off — without removing the control. And a rail that can *stop* a run degrades to a resumable checkpoint, never an error: a budget cutoff is a settled `budgetTruncated` turn or a typed `BudgetCutoffError` (worker), or a `cause: 'budget'` flag (orchestrator), each with a resume path and none routed to infra-retry.
- **Phase decision via a persisted marker.** `advance_phase`/`ask_human` record a terminal marker `{phase, kind}` written atomically with the gate packet or queued question it carries — the one cross-process channel for "which `phase.*` event to emit," read back through the pure, phase-matched `markerToEvent` (`phase-events.ts`) by both hosts. The lifecycle clears it deliver-before-clear (after the quiescent snapshot is durable, before any `gates_at` auto-cross). The subtlety that bit: the marker survives the crash window between the snapshot save and the clear, and a same-phase re-entry is *not* inherently harmless — a stale `flag` re-entered by an answer, or a stale `advance` by a reject, would replay the old decision and swallow the human's input. The **spent-marker guard** closes it: on entry the lifecycle reads the *restored snapshot* and clears the marker iff it resumed at that marker-phase's own gate/flag-wait (the transition already applied); a crash *before* the transition restores at a prior state, so the live marker still replays as intended. The same marker is honored in the driver's failure **catch**, before any classify/retry/flag: a terminal marker written just before a post-`advance_phase`/`ask_human` stream throw is this turn's real decision (markers are phase-scoped), so honoring it there strengthens first-terminal-wins instead of letting a late infra error overwrite a recorded decision.
- **View-time color.** Log files are plain text always — they are the inspectable-without-duet artifacts. One palette (`colorize.ts`), applied only where a human is watching. The `⏳` heartbeat / `⋯` activity / `✗` error markers are a producer→colorizer contract: the producer writes the leading marker into the plain-text line, the colorizer keys on it (dimming `⏳`/`⋯`, reddening `✗`).
- **Atomic writes** (temp + rename) for everything crash recovery reads: `state.json`, `machine.json`.

## XState usage

The machine is deliberately small; the discipline is in what it refuses to represent:

- **Two event vocabularies, kept disjoint.** A phase state completes by emitting an internal `phase.advance`/`phase.flag`; a gate or flag-wait transitions only on a `human.*` authority event (approve, reject, answer). A gate has no `phase.*` handler and a phase state has no `human.*` handler, so `advance_phase` can *park* a run at a gate but can never *cross* it — that's a property of the vocabulary, not a prompt. No tool emits `human.*`, in-process or over the MCP boundary, so gate-skipping stays *unrepresentable* even for an out-of-process orchestrator.
- **Tags are API, not decoration.** `quiescent` = the lifecycle may persist a snapshot here (no live actors); `gate` / `flag-wait` / `phase` drive CLI copy and event validation. The tag sets are pinned exactly by tests (`tests/machine.test.ts` coherence suite) because a mis-tagged state would persist a snapshot that restore cannot resume.
- **Persist only at quiescence; restore never resumes an invoke.** Mid-phase crash recovery re-enters the phase loop; position truth is the JSONL transcripts. `state.json.machineState` is display-only — reconstructing machine state from it is the anti-pattern, and the spent-marker guard (below) keys off the *restored snapshot* (`machine.json`), never that mirror, for exactly this reason.
- **The snapshot is typed at the store boundary** (`Snapshot<unknown>` in `run-store.ts`), so hydration is `createActor(machine, { snapshot })` with no casts anywhere.
- **Context is input, not state** (`runId`, `cwd`, `hasSpec`). Anything that changes during a run lives on disk, owned by the run store.
- **`snapshot.can(event)` before side effects** — the CLI validates a gate decision against the restored machine before staging input, so a wrong flag errors instead of silently no-oping.
- **Test through `machine.provide`** — script the actor, never reach into machine internals.

## Testing strategy

Standalone `tests/`, not co-located: the test surface is the public interface, and the interfaces here span process boundaries — co-located files invite testing internals. Fixtures compose via `test.extend` (`tests/helpers/fixtures.ts`: tmp project dir → run on disk → fake workers); the scripted statechart lives in `tests/helpers/scripted-machine.ts`.

- Behavior through real interfaces: tool handlers, store functions, machine events, rendered strings.
- Fake only at the seams table above. Filesystem and git run real, in tmpdirs. Time via fake timers (heartbeats only).
- Never mock our own modules; when tempted, write a third adapter for the seam instead.
- Deliberately untested: view glue (its designed failure mode is already "degrade to a one-line note") and the thin codex SDK wrapper.
- `tests/snippets.test.ts` guards the hand-edited `snippets.toml` and cross-checks every snippet key the orchestrator prompts name — a broken library fails a five-second test, not a $90 run.
- `tests/skill.test.ts` guards the shipped concierge skill the same way: every duet verb and flag the skill names must exist on the imported command table, and read verbs must be the only pre-approvals — a renamed flag fails in five seconds, not in a phone session.

## Build & publish

Dev has no build step: Node 24 runs `.ts` directly, and the global `duet` command (`pnpm add -g .`) is a symlink whose bin points at `src/cli.ts` — every edit is live. Publishing is the one place this cannot hold: Node refuses type stripping for files inside `node_modules` (`ERR_UNSUPPORTED_NODE_MODULES_TYPE_STRIPPING`), so an npm-installed copy must ship JavaScript.

The split lives in `package.json`: `bin` → `src/cli.ts` for dev; `publishConfig.bin` → `dist/cli.mjs`, rewritten by pnpm only in the published tarball. `pnpm build` (tsdown, config in `tsdown.config.ts`) bundles `src/` into `dist/cli.mjs` with dependencies external and publint validating the result; `prepack` chains typecheck → tests → build, so a tarball can't be cut from a broken tree. `dist/` is gitignored and never used in dev.

Facts the setup depends on: `snippets.toml`, `skills/`, and `prompts/` are listed in `files` because all are resolved package-relative at runtime — `src/snippets.ts` reads `snippets.toml`, and `src/orchestrate.ts` feeds `prompts/orchestrator-identity.md` to the launcher (`src/` and `dist/` both sit one level below the root, so the `import.meta.url + '..'` resolution survives bundling; a missing `prompts/` would point the launcher at an unshipped file). The `_drive` respawn uses `process.argv[1]`, so it is agnostic to which entry was invoked, and the `duet orchestrate` launcher self-references the same way — both the process spawn and the `_mcp` server it bakes into the session use `process.execPath` + `process.argv[1]`, never a bare `duet` on `PATH`, so the session attaches the same duet that launched it. `private: true` stays as the guard against accidental publish; the `license` (MIT), `author`, and `repository` fields are now set, so removing `private` and checking the npm name is the deliberate remaining step when publishing becomes real.

## Condensed lessons

1. **Don't approximate judgment with mechanism.** Every pre-pivot compensation (schema fields, disagreement string-matching, caps as exits) existed because code lacked judgment. When tempted to parse LLM output for meaning, hand the judgment to the orchestrator; keep code for structure.
2. **Guarantee in structure, steer in text.** The same rule usually needs both halves: the cap refuses in the handler *and* the refusal names the legal next moves.
3. **Soft constraints want warn-once-then-allow.** A hard block on a usually-wrong action recreates the dumb-router trap; one steering refusal plus a deliberate identical retry keeps judgment in charge and leaves the choice in the transcript.
4. **Infrastructure parameters leak into product scope unless named.** The per-turn budget rail shaped a descope decision before the prompts said "budget is per-turn; never shrink scope for it" (Q19).
5. **Fixed value + deterministic consumer → config; anything soft → prose.** The frontmatter boundary rule: a deterministic consumer of a soft value *enforces* errors that judgment would have resolved.
6. **Long silent operations need heartbeats.** Non-streaming worker turns run 30+ minutes; a silent pane reads as a hang.
7. **Every stop needs a next command.** A run state the human can see but not act on is a bug — the crash-without-question gap was exactly this, and the fix (crash = flag) is now a pattern. A stop the run can recover from *itself* goes one step further: model it as a resumable checkpoint with a resume path (the budget cutoff), not an error routed to retry/`errorClass`.
8. **A read across several surfaces wants one shared rule, not one per surface.** `doctor`, the heartbeat, and crash classification read the same transcripts; the "is this error still live" staleness bug recurred because two places judged it independently. Collapsing them onto `currentTerminalError` was the fix — convergence, not a local patch.
