# Engineering notes — the codebase's mental model

How the implementation is shaped and why. Companion to `automation-design.md` (the system design: roles, phases, gates, policies). Read that before changing *behavior*; read this before moving *code*. Audience: an engineer — human or agent — touching `src/` for the first time.

## The trust gradient

One sentence holds the architecture: **deterministic code constrains an intelligent orchestrator, which commands two workers — trust flows one way.**

```
statechart (when)   →   tool handlers (what's allowed)   →   prompts (judgment)
hard guarantee          enforced rail                        steerable
```

Placement rule for any new rule: violating it would break the human's authority or the run's integrity → statechart or tool handler. Violating it would merely waste tokens or quality → prompt. The design history (automation-design §"Design history") is one long lesson about this line — the pre-pivot router put *judgment* in code (string-matching disagreements, caps as exit rules) and failed; the pivot moved judgment up into an LLM and moved *guarantees* down into structure. Never drift back: no code that parses worker prose for meaning, no prompt that is the only thing between an agent and a gate.

## Module map

The trust gradient grounded in code. Modules cluster by subsystem; each line names what the module owns and the one fact that bites, *italicized* where it's load-bearing. Deep per-module mechanics live in the source — this is the way in, not a catalogue.

**The arc & the statechart**

- `src/phases.ts` — the **workflow registry** (`WORKFLOWS`): the arcs as data (ordered phases, gates + copy, caps, budgets, snippet sets, consultant checkpoints). *The single source.* Phase identity is workflow-scoped — `phaseSpec(workflow, phase)` is the one per-phase lookup, no global `PHASE` map (see the pattern below), and every topology/policy helper takes the arc — including `consultantCheckpointLive`, the one gateless gate *both* the orchestrator briefs and `list_snippets` read, so a gateless run drops the same checkpoints — only the holding bet-audit `challenge` (`survivesGateless`), keeping the `generative` frame and the `backstop` — on both surfaces. `validateRegistry` guards the derivation at load.
- `src/harness/machine.ts` — the statechart, built per-arc by `machineFor`. *A phase emits `phase.*`; a gate crosses only on `human.*`.* `interactiveMachineFor` is the inert-`phaseDriver` variant the interactive host rests on.
- `src/harness/phase-events.ts` — the `phase.*` / `human.*` vocabularies, and the one place the persisted marker is read back as an event.

**Tools & prompts**

- `src/harness/tools.ts` — the host-neutral `KernelTool` registry and every protocol rail. *The deepest module: all enforcement here, no Agent SDK import.* `send_prompt` is host-switched — blocking headless, dispatch-and-collect (`check_turns`) interactive.
- `src/harness/orchestrator-prompts.ts` — the system prompt + per-phase entry/resume briefs + the steer block (governed by `prompting-and-tool-design.md`).

**The run loop & its hosts**

- `src/harness/host-runner.ts` — the **host-neutral phase run loop** (`runHostedPhase`) + the `PhaseHost` seam: the rails every host shares (entry replay, nudge-once, twice-ended, crash → flag + bounded auto-retry — default-on for new runs). Its adapters supply only what differs — how a turn is driven, classified, whether it retries.
- `src/harness/driver.ts` — the **in-process host**: one phase = one orchestrator SDK session (`streamTurn`). *`retryable: true` — the one headless place auto-retry runs.*
- `src/harness/stdio-host.ts` + `src/harness/mcp-server.ts` — the **out-of-process & interactive hosts**: the same kernel over stdio MCP (`duet _mcp`). The run-scoped server holds the single-writer lease (`mcp-owner.json`) and refuses once a run is no longer interactively hosted (the two-writer guards).
- `src/harness/turn-dispatcher.ts` — the interactive host's pending-turn engine (dispatch → settle → collect). *Non-throwing / total:* a faulting turn flips to a collectible `failed`, never strands a role `running`.
- `src/harness/lifecycle.ts` — the detached `_drive` process. *The only place machine actors run for real.* Owns `probeRunPosition` (where a run is), the `gates_at` auto-cross + spent-marker guard, the inline interactive crossings, and `freezeContractAt`.

**Workers, providers & roles**

- `src/providers/` — the worker seam: claude + codex adapters + factory, and the `WorkerProvider` contract (the resumable `WorkerTurn` vs the typed `BudgetCutoffError`). *Exactly two providers by design;* the claude provider's interactive TUI is a second *transport*, not a third provider.
- `src/roles.ts` — the **role-policy table**: the consultant's asymmetries as data; `workerRolesFor` (workers) vs `voicesFor` (+ orchestrator). A pure runtime leaf — the "deletion test" module: drop it and `role === 'reviewer'` checks regrow at N sites.

**Persistence & sessions**

- `src/run-store.ts` — `.duet/runs/<id>/` persistence, atomic. *`state.json` is a HINT — the transcripts are truth.* Home of the terminal marker, the interactive lease/markers, and the `gateAttended` / `budgetFor` resolvers. *Every mutation funnels through `mutate()`* (fresh-load → apply → save → re-apply to the caller's copy), so a CLI write can't clobber a live driver's concurrent saves.
- `src/steer-store.ts` — the human's mid-phase steer channel, its own module: file-per-steer staging under `steers/`, deliver-by-rename into `delivered/`. *Never `state.json`* (a CLI write there would race the live driver).
- `src/sessions.ts` — locating provider transcripts by exact session id. *The one module reaching outside `.duet/`* (what `--purge` deletes) — always an exact `<id>.jsonl` match, never a directory sweep.

**CLI, framing & status**

- `src/cli.ts` — command wiring only; parses under `import.meta.main` (so the table imports clean). Hides `_drive` and `_mcp`. *Genuinely thin:* the one command with real branching, `continue`, gathers facts and hands the decision to a pure planner, then just executes the returned action.
- `src/continue-planner.ts` — that planner: `(run state, restored facts) → ContinueAction`, no IO, exhaustively unit-tested (the `takeoverPlan` precedent generalized). It holds the crash-recovery / gate-decision / interactive-crossing logic that used to be tangled into the CLI handler.
- `src/orchestrate.ts` — the `duet orchestrate` launcher (the run-scoped `_mcp`, the identity prompt, the one gate-safety `ask` rule). Also the warm-start/reconnect resume: `--resume-session <id>` opens the orchestrator on an existing session (`claude --resume`) with a transition kickoff, persisted as `interactiveOrchestratorSessionId` so a drop reconnects the same one. *That field is deliberately distinct from the headless driver's `orchestratorSessionId`* — the driver SDK-resumes the latter, so conflating them would resume a human TUI session over the SDK.
- `src/framing.ts` — the framing's journey (seed → editor → frontmatter parse); the machine/prose boundary rule lives here.
- `src/status.ts` — RunState + position → the status model and its renderers (text, `--json` verbatim, `--brief`). Pure; *the JSON schema is additive-only, pinned by test.*

**Health & supervision**

- `src/worker-health.ts` — the **pure** health substrate (taxonomy, `probeRole`, `retryDecision`). *Imports no fs/clock/lifecycle — purity is load-bearing,* or a `status → worker-health → lifecycle → status` cycle closes.
- `src/worker-activity.ts` — its pure sibling: the live-activity line from a transcript tail; liveness, not path-precision.
- `src/context-guard.ts` — the context-pressure substrate: the 75/85 bands, the transcript fill parser, the salvage-compact instructions. *Pure like `worker-health.ts`* — no fs/clock; the tail read lives in `sessions.ts`, the wiring in `harness/tools.ts` and the claude provider.
- `src/doctor.ts` — `duet doctor`'s composer; cli-only, so it may value-import `lifecycle` without re-closing the cycle `worker-health.ts` avoids.
- `src/stats.ts` — `duet stats`'s effort model from the voice logs; the log-reading sibling of `doctor` (status never reads logs), fail-soft.

**View glue & evidence** — best-effort, never allowed to affect a run

- `src/colorize.ts`, `src/tmux-view.ts`, `src/notify.ts` — view glue, deliberately untested.
- `skills/duet-concierge/` — the shipped concierge skill (zero runtime; coherence-guarded by `tests/skill.test.ts`).
- `src/spike/` — the substrate spike, the SDK pause/resume repros, and the context-accuracy probe (re-run on a CLI upgrade); executable evidence, not tests — do not modernize.

## Seams

A seam = a place behavior swaps without editing code in place. The deliberate ones — tests fake **only** at these:

| Seam | Adapters | Why |
|---|---|---|
| `WorkerProvider` (`providers/types.ts`) | claude, codex, tests' `FakeWorker` | Role–provider decoupling (design) doubles as worker scripting (tests) |
| `RunOrchestratorTurn` (`harness/driver.ts`) | Agent SDK, scripted sessions | The in-process SDK boundary; a fake session receives the real tool handlers and may invoke them |
| `Orchestrate` (`harness/stdio-host.ts`) | scripted client (tests), interactive CC session | The out-of-process sibling of `RunOrchestratorTurn` — drives the kernel over a real stdio MCP boundary; the seam the interactive orchestrator runs through |
| `phaseDriver` actor (`harness/machine.ts`) | `runPhase`, `runPhaseOverStdio`, inert (`interactiveMachine`), `machine.provide` scripts | XState's own substitution mechanism; the in-process, stdio, and interactive hosts all plug in here, so the lifecycle is reused unchanged |
| `PhaseHost` (`harness/host-runner.ts`) | in-process driver (`makeInProcessHost`), stdio host (`makeStdioHost`), tests' scripted host | One level *under* the two real `phaseDriver` adapters: the shared phase run loop (`runHostedPhase`) drives a `PhaseHost` whose only variation is how a turn is driven, how a failure is classified, and whether it retries — so the rails are tested with a 5-line fake session, not a full SDK or a real `_mcp` subprocess |
| Environment (`$EDITOR`, config path, notify fn, `ClaudeLauncher`, `isTTY`, connectivity `fetch`, transcript `home`) | real / stubbed / recording | OS boundaries — incl. the `duet orchestrate` process spawn (`orchestrate.ts`, a recording launcher in tests), the non-TTY write path, and the doctor connectivity / transcript reads |
| `PaneController` (`providers/pane.ts`) | `TmuxPane`, tests' `FakePane`, owned-pty (later) | The injection/process sub-seam *inside* the interactive claude transport — keeps the transcript parser transport-independent so owned-pty swaps in unchanged (`interactive-transport.md`) |

Rule: a new abstraction earns a seam only when a second adapter exists or a test needs one (`PaneController` earned both — a test now, pty later). Otherwise call directly — pass-through layers get deleted on sight.

## Patterns that carry the design

- **Workflow registry.** Adding or tuning a phase is one row in `phases.ts`; adding an *arc* is one entry in `WORKFLOWS` (mostly data). A `Record<PhaseName, …>` appearing anywhere else means the field belongs in the registry.
- **Workflow-scoped phase identity.** A phase is identified by `(workflow, name)`, not a global name — so both arcs name their build phase `implement` and their finish phase `finish`, and their specs still differ per arc. `phaseSpec(workflow, phase)` is the one per-phase lookup; there is no global `PHASE[name]` map (a shared name would collapse it to one arbitrary entry), and `validateRegistry` enforces uniqueness *within* a workflow. The run-state maps stay `Record<PhaseName, …>` — a run lives on one arc and only ever keys its own arc's phases, so the shared-name union is unambiguous per run. Every topology/policy helper takes the arc (`machineFor(workflow)`, `phasesOf`, `phaseOfGateState`, `isPostHandoffPhase`, the consultant cluster); a phase accepted from outside (`_mcp`) is validated against the run's arc. The mistake to avoid: a bare-name lookup that assumes global uniqueness (the pre-refactor `PHASE[name]` / `workflowOfPhase`).
- **Role policy as data; worker vs voice.** A role's asymmetries live once in `roles.ts`'s table, never as `role === 'reviewer'` checks sprinkled across the hosts (the consultant added a third role and the table absorbed it — the deletion test: drop the table and the conditionals re-grow at N sites). The split that threads every enumeration surface: **workers** (the bound makers/reviewers — `workerRolesFor`) drive routing, send-prompt enums, branch-fixing, orphan recovery; **voices** (`voicesFor` = orchestrator + workers) drive display/log/health, which must keep the orchestrator a reviewer-routing surface would drop. Picking the wrong one is a silent bug — a voice surface routed through `workerRolesFor` loses the orchestrator; a worker surface through `voicesFor` tries to prompt it. And the optional consultant is **default-off byte-for-byte**: absent its binding, every surface (config, tool schema, the snippet library, the phase briefs) reads exactly as before — the run-facing guarantee, tested at the tool altitude, not just the helper's.
- **Rails as named, ordered tool results.** Every rail is a small named function `(input, ctx) → refusal | null` that *refuses with steering text*: template-economy warn-once-then-allow, review-round backstop caps, branch-fixed-after-first-prompt, the phase-exit gate, the contract/verify checkpoints. A handler composes its rails with `firstRefusal` (first to refuse wins), and the **order is load-bearing** — the same-role-in-flight check precedes orphan-recovery, so a live turn refuses before the recovery seam fires its one side effect. A rail isn't real until its result text tells the orchestrator what to do instead (prompting doc, convention 5). Two supports keep them clean. (1) Result-envelope **builders** (`block` / `ok` / `error` / `refuse`) collapse the repeated `CallToolResult` shapes onto one taxonomy and make a text-less refusal a *compile* error — keeping a rail's `refuse` typed apart from a worker/tool `error`. (2) Two **boolean host oracles** on the rail context, `inFlight(role)` and `orphanedOnDisk(role)`, are the single place the blocking-vs-async host divergence lives, so a rail reads one registry and never branches on host. The invariant the oracles must preserve: the **phase-exit gate is async-host-only** — on the blocking host a `send_prompt` runs to completion before any terminal call, so no uncollected turn can exist, and the gate must not let the in-memory in-flight set (which the same-role guard legitimately reads there) strand a phase exit.
- **Host-neutral kernel, three hosts.** The tool surface is one `KernelTool` registry (`tools.ts`), independent of any SDK; thin adapters host it — the in-process Agent SDK server (`driver.ts`'s `toSdkTools`), the standard single-phase stdio MCP server (`mcp-server.ts`, reached via `duet _mcp <runId> <phase>`), and the run-scoped phase-less server the interactive orchestrator connects to (`duet _mcp <runId>`). The same handlers, rails, and steer delivery serve all, so an out-of-process orchestrator behaves identically to the in-process one — proven by a parity test driving a real `_mcp` subprocess. The single dependency this added, `@modelcontextprotocol/sdk`, is pinned to the one version the Agent SDK already resolves, so every host shares one `CallToolResult` type.
- **Host-neutral run loop, two phase hosts.** A second host axis, distinct from the tool kernel above: the *phase run loop* (`runHostedPhase`, `host-runner.ts`) owns the four rails every host shares — entry marker-replay, nudge-once, the twice-ended flag (a second turn-end with no advance or question ⇒ the run is stuck), crash → flag + bounded auto-retry (default-on for new runs) — behind the small `PhaseHost` seam. Its two adapters are the `phaseDriver` actor's two real implementations: the in-process Agent SDK driver (`driver.ts`, `makeInProcessHost`) and the stdio MCP host (`stdio-host.ts`, `makeStdioHost`); each supplies only what legitimately differs — how a turn is driven (`HostedSession.driveTurn`), how a failure is classified (`classifyFailure`), whether it retries (`retryable`). The interactive run-scoped server is *not* a third here — its loop is the turn-dispatcher and the machine rests inert. The drift this collapsed: the stdio host had silently diverged to a cheaper crash classification and its own copy of the twice-ended / crash-flag wording — lesson #8 (one shared rule, not one per surface) applied to the host rails.
- **Interactive orchestrator host.** The human's own CC session is an orchestrator host over the attended arc, up to the workflow's handoff gate (full: through PLAN; rir: through Direction). Its server is *run-scoped*, not phase-scoped: a long-lived session can't keep swapping a static phase arg, so it resolves the live phase from disk per call and refuses once the run is no longer interactively hosted (`orchestrationHost`), closing the two-writer gap against a handed-off headless driver. The **rails-only cache** is the load-bearing tactic: the tool surface is rebuilt per call against *fresh* disk state — so a cross-process `duet continue` (a same-phase reject/answer) is seen immediately, not served stale — while only the per-phase rails (the same-role in-flight guard, the warn-once set) and providers are cached and rebuilt at a phase boundary. The machine rests inert (`interactiveMachine`): a non-quiescent phase-loop snapshot is a legitimate rest, and `crossInteractive` advances it inline (consume the marker's `phase.*` to reach the gate/flag, *then* apply `human.*` — a phase loop ignores `human.*`, only the gate/flag takes it) with no `_drive` until the handoff gate. Gate safety is structural as everywhere (`get_task`/`advance_phase` only park), plus one `ask` rule on `duet continue` that survives the human's bypass-permissions default — the colon form `Bash(duet continue:*)`, matching the concierge's read-verb rules.
- **Fire-and-collect worker turns (interactive host).** A CC assistant turn can't yield while a `tool_use` is outstanding, so a blocking `send_prompt` would freeze the session for the minutes a worker runs. On the interactive host an `async` dep flips `send_prompt` from blocking to *dispatch-and-return* and exposes `check_turns` to collect later — a host switch in the one registry, not a fork (the headless driver keeps the blocking path and never leaves a turn in flight). The `turn-dispatcher.ts` engine owns the pending-turn lifecycle (`dispatch` → settle → `collectReady`), running the *same* settle/render/heartbeat steps a blocking call does, spread across time; it reads fresh disk state at each step because it outlives the dispatching call. Three invariants carry it: (1) a **single-writer lease** (`mcp-owner.json`) fenced at two boundaries — `toolsFor`'s synchronous gate covers every dispatch-time write, the background settle keeps its own non-throwing lease check — so a superseded server's late settle writes nothing; (2) the background lifecycle is **non-throwing / total** on every exit: a faulting setup, launch, or settle flips the role to a collectible `failed` rather than stranding it `running` (which would hang `check_turns` / `status --wait` forever under AFK), and the heartbeat is always stopped; (3) a turn orphaned by a session quit is a **durable record with no live owner** — detected on disk, never from a transcript guess — which refuses a same-role resend and blocks phase-exit until `duet takeover` clears it. The slice ordering is load-bearing: the lease lands before the async writer, and the orphan invariant holds on disk from the writer's first commit, so every committed tree is self-safe under AFK. Test-verified; the live end-to-end and environment smoke tests are still pending.
- **AFK resilience — native-vs-own, two recovery planes.** The seams that keep the overnight window honest (the design + the verified Claude facts live in `docs/automation-design.md` §"Resilience for the AFK window" — single source of truth; this names where each seam is). The native stall-watchdog rides an env knob injected at all three Claude connections (`claudeExecaOptions`, `buildOrchestratorOptions`, the extracted `claudePaneLaunchCommand`) — Claude-only, never codex. The owned wall-clock backstop is `runWithWallClockDeadline` (`providers/wall-clock.ts`, Date-based, injectable `now`/`schedule`), wrapping both providers' run loops; the per-turn cap rides the `RunTurnOptions.timeoutMs` contract every provider honors. The honest accepted-abort split lives at the provider failure seam (`recoverClaudeFailure` + the pure `transcriptShowsPromptAccepted`; codex's `recoverCodexAbort` keyed on a `thread.started`-seen flag, never `onSessionId`) and settles through `settleTurn`'s `aborted` branch (no round, no consultant marker) — render stays presentational. The two planes diverge by *what fails*: a worker turn → orchestrator-routed via the result text; the orchestrator's own session → the mechanical `retryDecision` plane (`--retry-infra` governs, headless-only). The lifecycle owns caffeinate + the quiescence soft-fail (`lifecycle.ts`); the `/compact` reset rides the shared `settleTurn` path.
- **Context pressure — one scope gate, a high-water reading.** The design is `automation-design.md` §"Resilience for the AFK window"; the shape that bites when moving code: `contextPressureApplies` (`harness/tools.ts`) is the **one** scope predicate every consumer shares — the turn's context cap, the send-side rail, the footer's band marker — so the claude-persistent-headless scope can't drift per surface. Guards read the **high-water** fill since the last compact (`contextSafetyPercent`), never the display percent: an error turn can't lower the safety reading, a completed compact or session reset **clears** it rather than recording it (a compact turn's own usage measures the summarization request — the very size just removed), and a window change restarts the mark. `providers/wall-clock.ts` hosts both deadlines over one shared abort-once-then-drain race (`raceCut`); the context deadline samples the transcript tail, and an undefined sample never cuts. In the rail order, context pressure sits after orphan-recovery and before the review cap; a `/compact` is detected from the literal body, never the tag, and is the one send the emergency band lets through. The salvage ladder is blocking-host-only (the auto-retry human-present split), and every intervention is ledgered (`contextEvents`) through the same `settleTurn` path.
- **Cooperative pause.** `ask_human` persists the question at call time, the result text says "end your turn", the process exits at quiescence. Mechanical SDK interrupts corrupt resume — proven, with executable repros (`src/spike/repro-*.ts`).
- **Parallel worker turns.** The claude CLI runs MCP tools serially unless `annotations.readOnlyHint` is true (its scheduler's concurrency test for MCP tools — verified 2.1.175, undocumented; full note: `docs/prompting-and-tool-design.md` §Concurrency). `send_prompt` carries the annotation as a concurrency hint, not a purity claim, so independent turns to different workers overlap; the genuinely unsafe case — two concurrent turns into one session — is refused by the handler's same-role in-flight rail. If parallel sends stop overlapping after a CLI upgrade, re-verify that mapping first.
- **Staging handshake.** Human input crosses the CLI→driver process boundary via `stageHumanInput`/`consumeHumanInput`, consumed exactly once so a retried driver can't replay an answer. Known edge, chosen deliberately: a crash after consume loses the staged text — the crash question asks the human to re-supply, which beats risking double-delivery into a session that may have already used it.
- **Steer channel.** The human's mid-phase voice rides tool results: file-per-steer staging under `steers/` (never `state.json` — a CLI write would race a live driver's saves), delivery appended to every phase-continuing result, carry-forward through the next harness prompt otherwise. The crash trade inverts the staging handshake's: deliver-then-consume, so a crash *redelivers* rather than loses (a repeated instruction is benign where a lost one is not). Turn-ending results (advance recorded, question queued) never carry steers — guidance delivered into a dying turn lands and dies — and the delivery path is fail-soft: a steer bug must never corrupt a tool result.
- **Position probe.** Where a run is cannot be read from the snapshot alone — for a pure-headless run persistence is quiescence-only, so mid-phase the snapshot shows the *previous* stop. `probeRunPosition` joins driver liveness, the parked snapshot, and the state evidence the driver writes continuously (`phaseStarted`, `pendingQuestion`) into running / gate / flag / crashed / abandoned / done; a crashed position names how its dead crossing is re-uttered (`approve` / `answer`), so bare `duet continue` recovers every crash shape. The one non-quiescent snapshot a headless probe meets is a leftover **phase loop** (e.g. `implementLoop`) after an interactive→headless handoff — `crossInteractive` rests the machine at the next phase's loop, then `orchestrationHost` is cleared — so the headless branch maps a `<phase>Loop` value to that phase (a live driver → `running` there, a dead one → a mid-phase `crashed` resumed from the loop snapshot); without it a handed-off mid-implement run misreported against the entry-phase fallback. The `abandoned` short-circuit comes first: `duet abandon` kills the driver and writes a marker, so a deliberate stop reads as abandoned rather than crashed — and stays reversible, since `continue` clears the marker and the transcripts are kept. Steer gating, crash recovery, and the status model all derive from this one probe.
- **Crash = flag.** Any infrastructure failure lands the run on an actionable queued question, never a silent state; a question the orchestrator already queued is never overwritten by a crash question. The stdio host shares this across the process boundary: a dead `_mcp` peer becomes the same persisted question, so crash = flag survives the boundary, and a silent turn gets one nudge before flagging. Parity is structural, not duplicated — the nudge-once, twice-ended, and crash-flag rails live once in the shared run loop (`host-runner.ts`), so both hosts emit the one wording.
- **Infra classification & bounded auto-retry (default-on for new runs).** One pure rule decides "is this run broken right now": `currentTerminalError` — a classified terminal error that is recent *and* not superseded by later activity — shared by the `crashed` verdict and the driver's infra classification so the two can't drift. The retry *policy* (`retryDecision`) is consulted in one place — the shared run loop's catch — but gated on the host's `retryable` posture, which only the in-process driver sets; the classification *taxonomy* reaches both hosts as their `classifyFailure` capability (the in-process driver's staleness-aware `classifyInfraError`, the stdio host's bare `classifyError` — it does not retry, so it needs no transcript refinement). Only the retryable headless path acts on a class (a human is present on the interactive host, so it classifies and hands back). Default 3 for a new run (materialized at `createRun`, an old/absent budget off byte-for-byte); transient classes back off and retry, `auth` retries once then escalates as `login-required`, the rest flag; exhaustion always flags.
- **Opt-in rails, safe defaults.** Run-policy resolvers read run state against the registry and fall back to legacy behavior when the knob is absent: `gateAttended` (absent `gatesAt` ⇒ attend all; an empty `gatesAt: []` is attend-*none*, distinct from absent) and `budgetFor` (absent `budget` ⇒ no caps; off ≡ absent, never `0`). `implementerModelFor` (`config.ts`) is the same shape for a *binding* rather than run-policy — absent `impl` ⇒ the implementer's base model in every phase, present ⇒ a distinct claude model for phases after the handoff gate; applying it *replaces* the implementer binding rather than mutating it, since an un-overridden `bindings.implementer` is still the shared `DEFAULT_BINDINGS` object. The defaults are friction-free — the PR auto-opens (Open-PR pre-authorized by default, the `overnight` posture), budget caps are off — without removing the control. The **gateless** posture (`state.gateless`, absent ⇒ off byte-for-byte) rides the same shape over two axes: it materializes `gatesAt: []` and narrows the consultant by *kind* through the `consultantCheckpointLive` predicate — dropping only the holding bet-audit `challenge`, keeping the non-holding `generative` frame and the `backstop` — changing *what produces holds*, not the severity-hold mechanism, which is untouched (the lone exception is `enterAfk`'s one backstop-preserving refusal). And a rail that can *stop* a run degrades to a resumable checkpoint, never an error: a budget cutoff is a settled `budgetTruncated` turn or a typed `BudgetCutoffError` (worker), or a `cause: 'budget'` flag (orchestrator), each with a resume path and none routed to infra-retry.
- **Phase decision via a persisted marker.** `advance_phase`/`ask_human` record a terminal marker `{phase, kind}` written atomically with the gate packet or queued question it carries — the one cross-process channel for "which `phase.*` event to emit," read back through the pure, phase-matched `markerToEvent` (`phase-events.ts`) by both hosts. The lifecycle clears it deliver-before-clear (after the quiescent snapshot is durable, before any `gates_at` auto-cross). The subtlety that bit: the marker survives the crash window between the snapshot save and the clear, and a same-phase re-entry is *not* inherently harmless — a stale `flag` re-entered by an answer, or a stale `advance` by a reject, would replay the old decision and swallow the human's input. The **spent-marker guard** closes it: on entry the lifecycle reads the *restored snapshot* and clears the marker iff it resumed at that marker-phase's own gate/flag-wait (the transition already applied); a crash *before* the transition restores at a prior state, so the live marker still replays as intended. The same marker is honored in the driver's failure **catch**, before any classify/retry/flag: a terminal marker written just before a post-`advance_phase`/`ask_human` stream throw is this turn's real decision (markers are phase-scoped), so honoring it there strengthens first-terminal-wins instead of letting a late infra error overwrite a recorded decision.
- **View-time color.** Log files are plain text always, and the stored timestamp is always raw UTC ISO — they are the inspectable-without-duet artifacts (so is `status --json`); only the *view* localizes. One palette (`colorize.ts`), applied only where a human is watching. The `⏳` heartbeat / `⋯` activity / `✗` error markers are a producer→colorizer contract: the producer writes the leading marker into the plain-text line, the colorizer keys on it. The `⏳` heartbeat and `✗` error are styled in place (dim, red); the `⋯` activity line is **promoted** to `[read|edit|search|run] <subject> <HH:MM>` — action tag in the role's color, subject secondary, the action's local time dimmed (deliberately louder than the content-free heartbeat). Header stamps render a local clock; the activity line trails the action's wall-clock time rather than a relative age, since a line is colorized once as it streams (a relative age would read ~0 and freeze). `timefmt.ts` holds the pure local-time helpers; the human-text renderers in `status.ts`/`doctor.ts` localize too, while `--json` stays UTC.
- **Atomic writes** (temp + rename) for everything crash recovery reads: `state.json`, `machine.json`.

## XState usage

The machine is deliberately small; the discipline is in what it refuses to represent:

- **Two event vocabularies, kept disjoint.** A phase state completes by emitting an internal `phase.advance`/`phase.flag`; a gate or flag-wait transitions only on a `human.*` authority event (approve, reject, answer). A gate has no `phase.*` handler and a phase state has no `human.*` handler, so `advance_phase` can *park* a run at a gate but can never *cross* it — that's a property of the vocabulary, not a prompt. No tool emits `human.*`, in-process or over the MCP boundary, so gate-skipping stays *unrepresentable* even for an out-of-process orchestrator.
- **Tags are API, not decoration.** `quiescent` = the lifecycle may persist a snapshot here (no live actors); `gate` / `flag-wait` / `phase` drive CLI copy and event validation. The tag sets are pinned exactly by tests (`tests/machine.test.ts` coherence suite) because a mis-tagged state would persist a snapshot that restore cannot resume.
- **Persist only at quiescence; restore never resumes an invoke.** Mid-phase crash recovery re-enters the phase loop; position truth is the JSONL transcripts. `state.json.machineState` is display-only — reconstructing machine state from it is the anti-pattern, and the spent-marker guard (below) keys off the *restored snapshot* (`machine.json`), never that mirror, for exactly this reason.
- **The snapshot is typed at the store boundary** (`Snapshot<unknown>` in `run-store.ts`), so hydration is `createActor(machine, { snapshot })` with no casts anywhere.
- **Context is input, not state** (`runId`, `cwd`, `hasSpec`). Anything that changes during a run lives on disk, owned by the run store.
- **`snapshot.can(event)` before side effects** — the CLI validates a gate decision against the restored machine before staging input, so a wrong flag errors instead of silently no-oping.
- **Test through `machine.provide`** — script the actor, never reach into machine internals.

## Testing strategy

Standalone `tests/`, not co-located: the test surface is the public interface, and the interfaces here span process boundaries — co-located files invite testing internals. Fixtures compose via `test.extend` (`tests/helpers/fixtures.ts`: tmp project dir → run on disk → fake workers); the scripted statechart lives in `tests/helpers/scripted-machine.ts`.

- Behavior through real interfaces: tool handlers, store functions, machine events, rendered strings.
- Fake only at the seams table above. Filesystem and git run real, in tmpdirs. Time via fake timers (heartbeats only).
- Never mock our own modules; when tempted, write a third adapter for the seam instead.
- Deliberately untested: view glue (its designed failure mode is already "degrade to a one-line note") and the thin codex SDK wrapper.
- `tests/snippets.test.ts` guards the hand-edited `snippets.toml` — every snippet key the orchestrator prompts name, plus that the `{{lessons_dir}}` references resolve to a shipped vendored file and no personal path leaks back in — so a broken library or a missed re-vendor fails a five-second test, not a $90 run. The same suite owns the override-layer contract: `snippets.ts` serves the *effective* library — the shipped base with optional user (`~/.config/duet/snippets.toml`) and project (`<cwd>/.duet/snippets.toml`) overrides merged over it (pure `mergeSnippetLayers`: per-key whole-body, fail-closed on an unknown key, byte-identical when no override file is present; `runtimeLibraryContext` is the one OS-home read, isolated across the suite via a `$HOME` setup file so no test picks up the developer's own override).
- `tests/skill.test.ts` guards the shipped concierge skill the same way: every duet verb and flag the skill names must exist on the imported command table, and read verbs must be the only pre-approvals — a renamed flag fails in five seconds, not in a phone session.

## Build & publish

Dev has no build step: Node 24 runs `.ts` directly, and the global `duet` command (`pnpm add -g .`) is a symlink whose bin points at `src/cli.ts` — every edit is live. Publishing is the one place this cannot hold: Node refuses type stripping for files inside `node_modules` (`ERR_UNSUPPORTED_NODE_MODULES_TYPE_STRIPPING`), so an npm-installed copy must ship JavaScript.

The split lives in `package.json`: `bin` → `src/cli.ts` for dev; `publishConfig.bin` → `dist/cli.mjs`, rewritten by pnpm only in the published tarball. `pnpm build` (tsdown, config in `tsdown.config.ts`) bundles `src/` into `dist/cli.mjs` with dependencies external and publint validating the result; `prepack` chains typecheck → tests → build, so a tarball can't be cut from a broken tree. `dist/` is gitignored and never used in dev.

Facts the setup depends on: `snippets.toml`, `skills/`, `lessons/`, and `prompts/` are listed in `files` because all are resolved package-relative at runtime — `src/snippets.ts` reads `snippets.toml` and resolves the `{{lessons_dir}}` token to the vendored `lessons/` methodology (`LESSONS_DIR`), and `src/orchestrate.ts` feeds `prompts/orchestrator-identity.md` to the launcher (`src/` and `dist/` both sit one level below the root, so the `import.meta.url + '..'` resolution survives bundling; a missing `prompts/` would point the launcher at an unshipped file). The `_drive` respawn uses `process.argv[1]`, so it is agnostic to which entry was invoked, and the `duet orchestrate` launcher self-references the same way — both the process spawn and the `_mcp` server it bakes into the session use `process.execPath` + `process.argv[1]`, never a bare `duet` on `PATH`, so the session attaches the same duet that launched it. `private: true` stays as the guard against accidental publish; the `license` (MIT), `author`, and `repository` fields are now set, so removing `private` and checking the npm name is the deliberate remaining step when publishing becomes real.

## Condensed lessons

1. **Don't approximate judgment with mechanism.** Every pre-pivot compensation (schema fields, disagreement string-matching, caps as exits) existed because code lacked judgment. When tempted to parse LLM output for meaning, hand the judgment to the orchestrator; keep code for structure.
2. **Guarantee in structure, steer in text.** The same rule usually needs both halves: the cap refuses in the handler *and* the refusal names the legal next moves.
3. **Soft constraints want warn-once-then-allow.** A hard block on a usually-wrong action recreates the dumb-router trap; one steering refusal plus a deliberate identical retry keeps judgment in charge and leaves the choice in the transcript.
4. **Infrastructure parameters leak into product scope unless named.** The per-turn budget rail shaped a descope decision before the prompts said "budget is per-turn; never shrink scope for it".
5. **Fixed value + deterministic consumer → config; anything soft → prose.** The frontmatter boundary rule: a deterministic consumer of a soft value *enforces* errors that judgment would have resolved.
6. **Long silent operations need heartbeats.** Non-streaming worker turns run 30+ minutes; a silent pane reads as a hang.
7. **Every stop needs a next command.** A run state the human can see but not act on is a bug — the crash-without-question gap was exactly this, and the fix (crash = flag) is now a pattern. A stop the run can recover from *itself* goes one step further: model it as a resumable checkpoint with a resume path (the budget cutoff), not an error routed to retry/`errorClass`.
8. **A read across several surfaces wants one shared rule, not one per surface.** `doctor`, the heartbeat, and crash classification read the same transcripts; the "is this error still live" staleness bug recurred because two places judged it independently. Collapsing them onto `currentTerminalError` was the fix — convergence, not a local patch.
