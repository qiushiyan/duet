# Future directions

The product-direction ledger: where duet goes next, what's shelved with a revisit trigger, and what was considered and declined. Companion to `docs/open-questions.md` (design decisions inside the current product); this doc tracks candidate *changes to what the product is*. Directions here were weighed against the product goals in `CLAUDE.md` and a user interview (2026-06-12) on actual usage: during AFK stretches the human is variously at the machine on other work, away with a phone, or asleep; mid-phase steering urges arise "sometimes"; at gates the packet text reads fine but composing substantial feedback in shell flags is the friction; remote reach matters.

**Standing constraint for anything remote:** duet never builds its own mobile/remote/notification layer. Either a direction works local-first, or it rides an existing product's remote surface (Claude Code remote control, Tailscale). Hand-rolled remote infrastructure is out of scope permanently.

## Active: Claude Code as the interaction layer

The gap it closes: the run can talk to the human (notification, packet, queued question) but the human couldn't talk back except at quiescent stops, from the terminal, through shell-flag strings. The interaction asymmetry was the biggest product gap after run-1. Two variants, sequenced deliberately: the concierge first, then the in-CC orchestrator — now built over the attended arc, with the first live session still ahead.

**B — concierge: built** (spec: `docs/specs/2026-06-12-concierge-package.md`; plan: `docs/plans/2026-06-12-concierge-package.md`). Three pieces ride existing seams, and Claude Code's native remote control, mobile push, and `/loop` supervision become duet's remote layer for free:

- `duet steer "<note>"` — the human's mid-phase voice: file-per-steer staging, delivered verbatim on the orchestrator's next phase-continuing tool result, carried into the next harness prompt when it misses the phase; at a quiescent stop it refuses toward that stop's own channel. Design detail: automation-design §"The steer channel".
- `duet status --json` — the status model under both renderers; the discriminated `stop` (running / gate / flag / crashed / done) carries the command that acts at each position, making the concierge's channel translation mechanical. Schema additive-only, pinned by test.
- `skills/duet-concierge/` — the shipped skill (repo-owned `skills/`, distinct from dev-time `.claude/skills/`; install by symlink or copy, the user's choice): verbatim relay, relay-not-fourth-engineer, turn-ending stop reports (what fires CC's mobile push), Sonnet-class session recommended in prose, never pinned. Read verbs are the only pre-approvals; the recommended `ask` rule on `duet continue*` makes a gate crossing take the human twice, phone included — even a rogue concierge cannot cross alone. Full envelope: it reads, steers, continues, and starts runs from dictation; only `takeover` stays terminal-only by nature.
- **run-operations (2026-06-19)** — supervising from outside gained the *health* half to match the position half: `duet doctor` (per-role verdict + connectivity), a health-enriched heartbeat, the signal-only `humanDecisions` / `cause` triage fields that make hold-vs-relay mechanical, a lean `status --brief`, a hardened non-TTY write path, and opt-in bounded infra retry. All ride existing seams; implemented and test-verified, awaiting the same first-live-session evidence (spec: `docs/specs/2026-06-19-concierge-run-operations.md`).

The harness gained no daemon, no server, no machine events — the concierge is interaction glue, killable without consequence (augment-never-lock-in holds). CC capabilities were verified against docs 2026-06-12: remote control (`/remote-control`, outbound-HTTPS, Mac stays awake — already true for any duet run), mobile push on turn-ending reports, `/loop`/Monitor supervision, permission `ask` prompts surfacing on mobile. Deferred utilities, until publishing turns real: a `duet skills install` convenience, a `.claude-plugin/plugin.json`, `duet runs --json`. One hazard to respect: never build on duet's own harness as a duet run — workers would edit the source live drivers re-spawn from. **What remains is evidence, not code: the first live remote session (phone + `/remote-control` + a real run), and the first overnight run that uses steers.**

**A — orchestrator in CC: built over the attended arc; live verification is what remains.** The orchestrator becomes an interactive CC session talking to the human directly — steering, triage, and gate conversation native, from anywhere.

The foundation (the host-neutral kernel) was proven first: the event-driven `phase.*`/`human.*` statechart, the persisted terminal marker with its crash-recovery guard, crash=flag, and nudge-once all behave identically whether the orchestrator runs in-process or over a real stdio MCP boundary — so re-homing the orchestrator into a separate process, once feared as the hard cost, is a settled property, not a rewrite. On that foundation the **user-visible orchestrator is now built** (spec/plan: `docs/specs/claude-as-orchestrator.md`, `docs/plans/claude-as-orchestrator-stage1.md`): the human's interactive CC session drives a run over FRAME → SPEC → PLAN and hands off to the headless driver at plan approval. The pieces — a `duet orchestrate` / `duet new --interactive` launcher that seeds a first turn so the session opens working rather than blank, a run-scoped MCP server the session connects to for the whole arc — on which `send_prompt` is fire-and-collect, so the session stays live and steerable while a worker turn runs instead of freezing inside the call for minutes (`docs/specs/2026-06-21-async-interactive-send-prompt.md`; test-verified, live end-to-end + smoke tests pending) — the launcher-fed `prompts/orchestrator-identity.md` system prompt (the role is wired at launch — there is no `/duet` slash command), inline `duet continue` crossings until the plan-gate handoff — all carry the read-only conductor role across compaction and reconnect. Its single safety setup is one `ask` permission rule on `duet continue` that survives the human's bypass-permissions default; no tool can emit `human.*`, so even a Bash-equipped session cannot cross a gate alone. Both ends of a run are now CC sessions: a shipped `skills/duet-frame/` skill authors the *framing* (a rough problem sharpened into a clean framing — real names, structure, gate posture — without changing intent or proposing solutions, then emitting the `duet new --interactive` command), and the interactive orchestrator drives it.

**What remains is evidence, not code** (mirroring B): the first live interactive orchestrator session — a real CC session driving a real run — the human's environment smoke tests, and a check that the launcher's seeded kickoff actually auto-submits in the interactive TUI (it's pinned in the launch argv but unverified live); nothing here has been exercised live yet. The direction also forecloses codex-as-orchestrator (Q17 — the launcher is claude-specific) and deepens CC dependency — a one-way door, acceptable under the standing constraint. **B (concierge) remains a strict stepping stone**: its primitives, steer, and relay discipline carry over, and living with B is the evidence for whether talking-to-the-orchestrator beats relaying by enough to fund living on A.

## Shelved — interesting, with revisit triggers

**Interactive Claude worker transport: production hardening** (built as a spike — [`docs/interactive-transport.md`](interactive-transport.md)). The implementer can already opt into driving the interactive `claude` TUI (`transport = "interactive"`) so its turns bill the flat subscription quota instead of the metered Agent-SDK credit pool — the cost driver (~$85 of the ~$93 first real run). The spike is tmux-driven, proven over fakes, with one live-auth gate still open. What's shelved is the *production* path, all of it carried by the same seam: an owned-pty transport behind `PaneController` (the transcript parser reused unchanged — no rework), failure isolation, phase-scoped pane reuse to kill the per-turn cold-start, a read-only variant so a claude reviewer can also bill the subscription, and the default flip. **Trigger:** the spike clears its live-auth gate *and* a real run strains the metered credit pool enough to be worth the native pty dependency — or the unattended heuristics (screen-polled readiness, fire-and-forget injection) actually misfire overnight.

**Project profile: a default-loaded pre-context (`.duet/` "CLAUDE.md-for-duet").** *The trigger fired and the scaffold half shipped (2026-06-16).* The copying it predicted showed up — two byte-identical planlab framings — so `duet new --template <name>` now seeds the editor draft from a project's own `.duet/templates/<name>.md` (bare `duet new` prefers `.duet/templates/default.md`); the common framing is written once and only the run-specific problem is filled in per run (`src/framing.ts`, `resolveTemplateSeed`). A template is **pre-baked framing** — parsed and archived like any framing — so the single-entry-seam property holds untouched and no new config surface appears. What stays shelved is the heavier **merge** model the original entry described: a durable profile auto-loaded into every run as pre-context, with the framing shrinking to the run-specific *delta* rather than carrying the whole (scaffolded) text each time, plus the optional layer where the orchestrator proposes profile additions at run end (human-gated like snippet edits). **Revisit trigger:** when the scaffold's copy-per-run cost starts to bite — a template drifts from the framings older runs archived, or you want one edit to reach every future run without re-templating. The templates already sit where a profile would grow, so the upgrade is a template gaining a `profile:` reference, not a rewrite.

**Eval/replay harness.** Re-run a recorded phase's orchestrator against scripted workers (voice logs replayed through the `WorkerProvider` seam; the SDK boundary is already injectable) and diff routing choices — snippets picked, adaptations, triage calls, convergence judgment — against the original. Turns prompt changes from faith into A/B evidence without $93 runs; Q13/Q19/Q20 all currently wait on real runs as the only eval. **Trigger:** the first prompt change that makes a run worse and we can't tell why. Compounds every other direction.

**Multi-run attention queue.** Worktrees already give parallel runs separate cwds; nothing manages the human side — which run needs me next, notification collisions, one combined gate queue. Concurrent worktree sessions are observed behavior (planlab Jun 10, `2c6a7f46` + `d531410e`). Strong synergy with the concierge: one chat session supervising N runs. **Trigger:** the first missed gate during parallel runs.

## Considered, not pursued

- **Local browser dashboard.** The original candidate; the interview removed its rationale — packet text reads fine (no rendering need), remote must not be hand-built (no act-from-anywhere), and the comfortable-input need is met by the concierge's chat box. What remains (buttons in a local browser) doesn't beat tmux + CLI. Recorded so it isn't re-proposed on the same grounds.
- **Arc presets** (lighter arcs for small fixes) — *shipped 2026-06-21 as the RIR arc*, reversing the earlier "duet stays epic-shaped" stance. A run now picks its arc; the lighter `research → implement` arc lives beside the full one (`docs/automation-design.md` §"Phases and gates").
- **Environment-proxy ergonomics** (pre-composed command requests at flags). Generic `ask_human` suffices at current run volume.
- **Third *specialist* worker role.** A third *maker* — a domain worker owning a slice of the build — still cuts against two-role legibility; specialization belongs in snippets, not roles. Distinct from the **consultant** (shipped 2026-06-22, `docs/automation-design.md` §"The three roles"): that is an *optional second reviewer voice*, not a specialist maker — additive, read-only, ephemeral, off by default. It widens the reviewer altitude; it does not add a maker, so the two-role legibility this entry protects still holds.
- **Multi-PR pivots inside one run.** Observed in the corpus, but "finish the run, start a new one" is cheaper than machine support.
- **OSS-ification.** Against the product goals; personal tool.
